# HRIV Image Workflows

Use this skill for image lifecycle work spanning backend processing and frontend
viewer behavior.

## Start Here

1. Read `references/image-workflows-map.md`.
2. Read `../../../docs/image-processing-lifecycle.md` for upload, worker, and
   processing status flow.
3. Read `../../../docs/image-metadata-and-versioning.md` before changing
   metadata, annotations, overlays, image replacement, or `If-Match`/ETag logic.
4. Use `$testing-image-processing` for end-to-end processing verification.
5. Use `$hriv-frontend-ui` for viewer UI details and `$hriv-backend-api` for
   router/schema/model changes.

## Fragile Contracts

- DZI tiles are generated by libvips/pyvips and served from `TILES_DIR`; keep
  tile paths, source-image records, and frontend tile URLs aligned.
- Background processing normally runs through arq/Redis with a backend
  `BackgroundTasks` fallback when enqueueing fails.
- `metadata_extra` replaces the full JSONB metadata object; use
  `metadata_extra_merge` for partial metadata updates.
- Canvas annotations and locked overlays must be cleared on image replacement
  because coordinates reference the old geometry.
- Viewer annotation and overlay saves must track the latest image version and
  avoid stale `409 Conflict` writes.
- Pyramidal metadata keys are re-derived on replacement only when the new file
  supports them.
